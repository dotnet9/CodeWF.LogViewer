# CodeWF.Log 设计

## 目标

日志只有两个明确用途：

1. 用户出口：控制台、Avalonia `LogView` 和通知，内容必须简洁、可理解、可操作。
2. 技术出口：日志文件，保留路径、参数、内部状态和完整异常堆栈。

普通 `Logger.*` 同时进入两个出口；`Logger.*ToFile` 只进入技术出口。调用方没有提供 `userMessage` 时，普通 `message` 作为用户内容，避免日志无故从界面消失。

## 核心模型

| 类型 | 职责 |
| --- | --- |
| `Logger` | 稳定的静态调用入口和进程级初始化 |
| `LoggerOptions` | 全局采集级别、控制台、队列和用户缓存配置 |
| `FileLogOptions` | 最终目录、文件大小、批量和刷新配置 |
| `LoggerHost` | 串行消费日志命令，协调文件、控制台和用户 Feed |
| `LogEvent` | 内部完整事件，包含技术消息、用户消息和异常 |
| `UserLogEntry` | 对外安全事件，只包含序号、时间、级别和用户消息 |
| `UserLogFeed` | 多订阅者广播和有限最近日志回放 |
| `ILogSink` | 技术输出端抽象 |
| `FileLogSink` | 持久文件流、定时刷新以及按日期/大小轮转 |
| `ConsoleLogSink` | 只输出用户内容 |

异常对象只存在于内部 `LogEvent`，不会进入 `UserLogEntry`。

## 数据流

```text
Logger.Info/Warn/Error(...)
        │
        ├─ 全局 MinimumLevel 过滤
        │
        ▼
有界 Channel<LogCommand>（满时等待）
        │
        ▼
LoggerHost 单消费者
        ├─ FileLogSink：技术消息 + 用户提示 + 异常堆栈
        ├─ ConsoleLogSink：用户消息
        └─ UserLogFeed：UserLogEntry
               ├─ LogView A：独立级别范围
               ├─ LogView B：独立级别范围
               └─ LogNotifications：只订阅新日志
```

单消费者保证同一进程内的文件、控制台和用户 Feed 顺序一致。每条日志使用单调递增序号，供多个视图去重和实现独立清空边界。

## 初始化与关闭

`Logger.Initialize` 一个进程生命周期只允许调用一次。初始化时：

- 校验枚举、队列、文件大小、刷新周期和时间格式。
- 将相对日志目录固定为绝对路径，防止运行期间工作目录变化影响落盘位置。
- 创建文件 Sink、控制台 Sink、用户 Feed 和后台消费任务。

`FlushAsync` 在队列中插入刷新命令，等待之前的日志完成处理。`ShutdownAsync` 关闭写端、排空队列、刷新并释放 Sink。关闭期间晚到的写入和 Flush 不会反向破坏业务退出流程。

## 文件策略

- 文件名：`Log_yyyy_MM_dd.log`，同日按大小追加 `_1`、`_2` 等序号。
- 文件句柄保持打开，避免每条日志重复打开文件。
- 达到 `BatchSize` 或 `FlushInterval` 时刷新。
- 文件使用 UTF-8（无 BOM）和 `FileShare.ReadWrite`，便于运行中查看。
- 单条日志超过文件上限时仍完整写入，不截断堆栈。
- 文件输出故障通过 `Debug` 和限频的友好控制台错误兜底，避免日志组件递归记录自身错误。

## 用户 Feed 与 LogView

`UserLogFeed` 保存有限数量的最近用户日志，并允许多个订阅者同时存在。订阅回调彼此隔离，一个展示方异常不会破坏日志主流程。

每个 `LogView` 维护自己的：

- `MinimumLevel` / `MaximumLevel`
- `MaxDisplayCount`
- `RefreshInterval`
- `TimestampFormat`
- 清空序号边界

过滤范围变化时只重建当前视图。`Clear` 不清空全局 Feed，也不影响其他视图。

## 通知

通知由 `LogNotifications` 的应用级附加属性在 `App.axaml` 中统一配置，`LogView` 不承担通知职责。`LogNotificationPresenter`：

- 只订阅启用后的新日志，不回放历史。
- 使用独立最小/最大级别范围。
- 支持 `InApp` 和 `DesktopWindow`。
- 一个应用只创建一个订阅，显示时动态选择活动窗口或主窗口作为宿主。
- 应用退出时取消订阅并关闭现有通知。
- 限制待处理通知数量，避免错误风暴无限占用内存。

## 级别语义

`Logger.MinimumLevel` 是全局采集门槛；视图和通知的最小/最大级别只是各自的展示过滤。推荐：

- `Debug`：技术诊断，通常配合 `DebugToFile`。
- `Info`：正常业务进度。
- `Warn`：可继续运行，但用户应检查或调整。
- `Error`：当前操作失败或部分能力不可用。
- `Fatal`：进程无法继续安全运行。

## 调用规范

```csharp
// 普通用户消息
Logger.Info("任务已保存。");

// 技术详情与用户提示分流
Logger.Error(
    $"读取任务失败：{taskFilePath}",
    ex,
    "任务无法打开，请确认文件由当前版本导出后重试。");

// 内部诊断，不进入控制台、UI 或通知
Logger.ErrorToFile("后台刷新任务异常退出。", ex);
```

跨进程场景遵循同一规则：异常在哪个进程发生，堆栈就留在哪个进程的日志文件；协议只传用户友好消息。
